# MVP 范围与优先级

> 核心原则：**先让数据跑起来，其他都是后话**
> 更新日期：2026-07-19

## 一、MVP 核心目标

**把用户现有的Excel数据高效导入、清洗、整合、管理起来，形成一个以专利为锚点的统一数据库，并支持AI批量处理字段。**

MVP不做：
- ❌ Wiki页面（先做详情页就行）
- ❌ 图谱可视化（先做关系列表）
- ❌ Docketing期限管理（后面再说）
- ❌ 费用管理（后面再说）
- ❌ 用户权限（单用户先不做）
- ❌ 工作流/审批（后面再说）
- ❌ 多用户协作（后面再说）
- ❌ 桌面应用打包（先Web本地跑通）

MVP必须做：
- ✅ 核心数据库设计（专利+产品+项目+标注+组织）
- ✅ Excel导入（字段映射、清洗、去重合并）
- ✅ 表格视图（类Excel，筛选排序批量编辑）
- ✅ 自定义字段系统
- ✅ AI字段（批量处理、API调用）
- ✅ 基础统计与导出
- ✅ 专利详情页

## 二、用户实际工作场景（来自用户描述）

### 2.1 组织架构

```
专利部门
├── 检索组
│   ├── 工程师A → 负责产品X → 积累了几千/上万条该品类专利数据
│   ├── 工程师B → 负责产品Y
│   └── ...
├── 分析组
└── 撰写组
```

**核心**：每个检索工程师手里有负责的产品，每个产品有大量专利数据（几千~上万条）。

### 2.2 现有数据情况

| 数据类型 | 来源 | 更新方式 | 字段示例 |
|---------|------|---------|---------|
| **基础专利数据** | 商业数据库下载 | 定期下载更新 | 申请号、公开号、标题、申请人、发明人、申请日、公开日、法律状态... |
| **分类标注** | 人工标注 | 静态积累 | 分类、子分类、技术领域 |
| **技术标注** | 人工标注 | 持续积累 | 技术问题、技术效果、技术方案 |
| **风险标注** | 人工标注 | 按需标记 | 是否有风险、风险等级、风险描述 |
| **项目关联** | 人工标注 | 按需标记 | 关联项目、关联模块、应用状态（已应用/风险相关/参考） |
| **法律状态** | 定期更新 | 下载更新 | 授权/实审中/驳回/视撤... |
| **其他自定义** | 随时新增 | 按需 | 几百个维度随时加 |

### 2.3 核心工作流

1. **数据更新**：从商业数据库下载最新Excel → 导入系统 → 去重合并 → 更新状态变化
2. **数据标注**：在表格里/详情页标注技术问题、技术效果、风险、项目关联等
3. **AI批量处理**：选中一批专利 → 用AI字段批量抽取/分类/打分
4. **筛选查看**：按产品/分类/风险/项目/法律状态等筛选查看
5. **统计导出**：按各种维度统计（发明人、申请人、法律状态、分类...）→ 导出Excel
6. **信息分享**：给研发看保护范围、技术主题等信息

## 三、MVP核心功能优先级

| 模块 | 功能 | 说明 |
|------|------|------|

| | 产品/产品线管理 | 产品作为数据组织的核心维度 |
| | 项目管理 | 简单的项目管理，专利可以关联项目 |
| | 组织/人员管理 | 工程师、负责人，谁负责哪些产品 |

| | 标签系统 | 分类、子分类、各种标签 |
| **数据导入** | Excel上传解析 | 支持.xlsx/.xls格式 |

| | 列配置 | 显示/隐藏列、调整列宽、列排序 |

| | 排序 | 多列排序 |
| | 单元格编辑 | 直接在表格里编辑单元格 |
| | 批量操作 | 批量选中、批量编辑、批量打标签、批量删除 |
| | 右键菜单 | 常用操作右键直达 |
| **AI字段** | AI字段定义 | 配置提示词、选择模型、定义输出类型 |
| | 手动触发计算 | 选中专利，手动触发AI字段计算 |

| | AI结果展示 | AI字段值和普通字段区分显示 |
| | 人工覆盖 | 可以修改AI计算的结果 |
| **统计导出** | 基础统计 | 按字段统计数量分布（法律状态、申请人、分类等） |
| | Excel导出 | 按当前筛选结果导出Excel |
| | 导出字段选择 | 选择导出哪些字段 |
| **专利详情页** | 基础信息展示 | 展示所有字段（系统字段+自定义字段） |
| | 字段编辑 | 在详情页编辑字段 |

## 四、核心数据实体（MVP版）

### 4.1 组织架构

```
组织单元 (OrganizationUnit)
├── 检索组
├── 分析组
└── 撰写组
    ↓
人员 (Person)  ── 属于 → 组织单元
    ↓
产品负责人关系 (ProductOwner)  ── 人员负责产品
```

### 4.2 产品/项目

```
产品线 (ProductLine)  ← 可选，作为产品的分组
    ↓
产品 (Product)  ── 每个产品有几千~上万条专利
    ├── 负责人（检索工程师）
    ├── 描述
    └── 专利集合
        ↓
项目 (Project)  ── 产品下的具体项目/模块
    ├── 所属产品
    ├── 描述
    └── 专利关联（带角色：核心/应用/风险/参考...）
```

**为什么产品是核心？**
- 用户明确说了："检索组的每一个工程师手里都会负责一些产品，不同产品有不同的专利"
- "积累很多几千条、上万条某一个品类的数据，都属于这样一个产品"
- 所以产品是数据的**主要组织维度**，而不是项目

### 4.3 专利

```
专利 (Patent)
├── 系统核心字段（固定列）
│   ├── 申请号 (application_number) ← 主键之一
│   ├── 公开号 (publication_number) ← 主键之一
│   ├── 标题 (title)
│   ├── 摘要 (abstract)
│   ├── 申请人 (applicant)
│   ├── 发明人 (inventor)
│   ├── 申请日 (filing_date)
│   ├── 公开日 (publication_date)
│   ├── 授权公告日 (grant_date)
│   ├── 法律状态 (legal_status)
│   ├── 专利类型 (patent_type: 发明/实用新型/外观)
│   ├── 申请国家/地区 (country)
│   ├── 所属产品 (product_id) ← 外键
│   └── 主分类号 (ipc_main, cpc_main)
│
├── 自定义字段（JSONB）
│   ├── 分类 (category)
│   ├── 子分类 (subcategory)
│   ├── 技术问题 (technical_problem)
│   ├── 技术效果 (technical_effect)
│   ├── 技术方案 (technical_solution)
│   ├── 是否有风险 (has_risk: boolean)
│   ├── 风险等级 (risk_level)
│   ├── 风险描述 (risk_description)
│   ├── 关联模块 (module)
│   ├── 应用状态 (application_status: 已应用/风险相关/参考)
│   ├── 保护范围说明 (scope_description)
│   ├── 备注 (notes)
│   └── ... 用户随时新增的字段
│
├── AI字段（单独存储，有版本和状态）
│   └── AI计算的字段值，带版本号、计算时间、模型信息
│
├── 关系
│   ├── 多对多 → 标签（Tag）
│   ├── 多对多 → 项目（Project），带角色
│   ├── 多对多 → 引用关系（Citation）
│   └── 多对多 → 同族（Family）
│
└── 元数据
    ├── 创建时间、更新时间
    ├── 数据来源（哪个批次导入的）
    └── 最后更新时间
```

## 五、MVP技术验证清单

在正式开发前，先验证这几个关键技术点：

- [ ] **SQLite + JSONB性能**：插入1万条数据、多字段筛选、排序，看响应时间
- [ ] **AG Grid万行数据**：1万行数据，滚动流畅度、筛选排序速度
- [ ] **Excel导入速度**：1万行Excel导入+解析需要多久
- [ ] **去重合并逻辑**：基于申请号+标题相似度的去重准确性
- [ ] **AI批量调用**：批量调用100条AI字段需要多久，token消耗多少
