# 项目风险与遗漏分析

> 状态：活文档，持续更新
> 更新日期：2026-07-19

## 一、最没有把握的事情（架构级风险）

### 1.1 轻量化 vs 功能丰富度的平衡

**风险描述**：
用户要求"开箱即用、不装Python环境、同事电脑直接跑"，但同时需要AI、向量检索、图谱、公式引擎等重型功能。如何在体积、启动速度、功能完整性之间找到平衡点，是最大的架构挑战。

**可选方案对比**：

| 方案 | 体积估算 | 启动速度 | AI能力 | 开发效率 | 风险 |
|------|---------|---------|--------|---------|------|
| PyInstaller打包Python后端 + 前端 | ~200-500MB | 3-10秒 | 最强（Python生态） | 高 | 体积大、启动慢、打包复杂 |
| Tauri + Node.js后端 | ~50-100MB | 1-3秒 | 中等（JS AI库有限） | 中 | AI生态弱，很多Python库用不了 |
| 纯前端 + WebAssembly | ~20-50MB | <1秒 | 弱（WASM限制多） | 低 | 性能差，功能受限 |
| 本地后端 + 云端AI | ~30-80MB | 1-3秒 | 最强（调用云端API） | 高 | 需要联网、数据安全顾虑 |

**推荐方向**：方案1（PyInstaller打包）为主，方案4（云端AI）为可选配置
- 理由：专利数据敏感度高，本地处理是底线；Python AI生态最成熟
- 体积和启动速度可以通过懒加载、模块拆分优化
- 云端AI作为可选开关，用户自己权衡

**验证优先级**：P0 - 第一天就要验证打包可行性和体积

---

### 1.2 自定义字段的查询性能

**风险描述**：
几百个自定义字段存在JSONB列中，当数据量到几万条、同时有多个筛选+排序条件时，查询性能会不会不可接受？

**风险等级**：中高

**应对策略**：
1. **分层存储策略**：
   - 系统核心字段（约30-50个）→ 独立列 + 索引
   - 常用自定义字段 → JSONB + GIN索引
   - 不常用字段 → JSONB普通存储
2. **字段"实体化"机制**：
   - 用户可以将高频查询/排序的自定义字段"升级"为实体列
   - 系统自动迁移数据 + 创建索引
3. **查询优化**：
   - 分页 + 懒加载
   - 复杂查询走异步 + 缓存
   - 预聚合表（看板统计用）

**验证优先级**：P1 - MVP阶段就要做性能测试

---

### 1.3 AI字段的可靠性与可预期性

**风险描述**：
LLM输出有随机性，作为"字段值"可能每次计算结果都不一样。用户对字段值的预期是"确定的、可追溯的"，AI字段会打破这个预期。

**风险等级**：高（产品体验风险）

**应对策略**：
1. **AI字段有明确的"AI标识"**：UI上和普通字段区分开，降低用户预期
2. **版本管理**：
   - 每次AI计算都记录：模型版本、提示词版本、温度参数、计算时间
   - 字段值变更历史完整保存
3. **缓存机制**：
   - 相同输入（专利数据+提示词+模型版本）缓存结果
   - 只有输入变了才重新计算
4. **人工确认机制**：
   - AI字段值可以被人工覆盖
   - 人工覆盖后标记为"人工确认"，不再自动更新
5. **可复现性优先**：
   - 默认用temperature=0
   - 尽量用结构化输出（JSON Schema约束）

---

### 1.4 专利业务深度的盲区

**风险描述**：
开发团队不是专利专家，业务模型设计可能有根本性缺陷，导致后期推倒重来。

**风险等级**：高（产品方向风险）

**应对策略**：
1. **领域驱动设计（DDD）**：
   - 先和专利专家充分讨论业务场景
   - 建立领域模型，定义核心概念和关系
2. **渐进式验证**：
   - MVP先做最通用的部分（专利基础信息+表格+导入）
   - 复杂业务模块（申请流程、Docketing）后期迭代
3. **配置化而非硬编码**：
   - 流程、状态、字段都可配置
   - 即使业务理解有偏差，用户也能自己调整
4. **找种子用户验证**：
   - 找1-2个专利工程师提前试用
   - 快速迭代修正业务模型

## 二、最大的遗漏（业务视角）

### 2.1 专利申请全生命周期管理

**之前的盲区**：
只考虑了"已授权专利的管理"，但专利部门80%的工作在申请过程中。

**完整生命周期**：

```
发明提案 → 交底书 → 内部评审 → 委托代理 → 申请提交
   ↓
受理 → 初审 → 公开 → 实审 → 审查意见答复 → ...
   ↓                    ↓
  授权                  驳回
   ↓                    ↓
年费维护              复审/诉讼
   ↓
无效/转让/许可/质押
```

**每个阶段涉及**：
- 不同的文档（交底书、审查意见通知书、答复意见、证书...）
- 不同的期限（优先权、答复、年费...）
- 不同的责任人（发明人、专利工程师、代理人、律师...）
- 不同的费用（官费、代理费、翻译费...）
- 不同的状态流转规则

**设计原则**：
- 状态机驱动（每个专利类型有自己的状态流转图）
- 阶段可配置（不同国家、不同类型的专利流程不一样）
- 时间线视图（一目了然看到全过程）

---

### 2.2 跨国专利的复杂性

**之前的盲区**：
把"同族"想得太简单，只是"一组相关专利"。实际上跨国专利管理非常复杂。

**核心概念**：

| 概念 | 说明 | 重要性 |
|------|------|--------|
| **优先权** | 首次申请后12个月内可主张优先权，视为同一天申请 | ⭐⭐⭐⭐⭐ |
| **PCT国际申请** | 一个申请指定多个国家，30个月后进入国家阶段 | ⭐⭐⭐⭐⭐ |
| **巴黎公约途径** | 直接在各国申请，主张优先权 | ⭐⭐⭐⭐ |
| **地区专利** | EP（欧洲）、OA（非洲）、AP（东盟）等，一次申请覆盖多国 | ⭐⭐⭐ |
| **同族类型** | 简单同族、复杂同族、延伸同族、专利家族 | ⭐⭐⭐⭐ |
| **分案申请** | 从原申请中分出一个新申请，保留原申请日 | ⭐⭐⭐ |
| **继续申请** | 美国特有制度，Continuation / CIP / Divisional | ⭐⭐⭐ |

**管理难点**：
- 优先权链追踪（一件专利可能有多层优先权）
- 国家阶段进入期限管理（错过就进不去了）
- 同族专利的法律状态联动（一个国家授权不代表其他国家也授权）
- 费用的国别差异（各国官费、年费完全不同）

---

### 2.3 组织角色与工作流

**之前的盲区**：
默认是"一个用户"在用，实际上是跨部门协作系统。

**核心角色**：

| 角色 | 主要工作 | 系统使用频率 |
|------|---------|-------------|
| **专利工程师/代理人** | 日常操作主力，全功能 | 每天 |
| **研发人员** | 提交发明提案、提供技术支持 | 偶尔 |
| **产品/项目经理** | 项目维度专利布局、风险跟踪 | 每周 |
| **法务/合规** | 风险评估、合同审核、诉讼支持 | 按需 |
| **部门负责人** | 看板、报表、决策 | 每周/每月 |
| **管理层** | 高层视角、战略决策 | 每月 |

**权限模型**：
- RBAC（基于角色的访问控制）
- 数据范围权限（只能看自己负责的/本部门的/全部）
- 字段级权限（敏感字段只有特定角色能看）
- 操作权限（查看/编辑/删除/审批）

---

### 2.4 Docketing（期限管理）

**之前的盲区**：
完全没提期限管理，这其实是专利部门的"生命线"。

**为什么重要**：
- 错过期限 = 专利失效 = 巨大损失
- 一个专利在生命周期内有几十个期限节点
- 跨国专利的期限计算非常复杂（各国法规不同、节假日顺延）

**核心功能**：
1. **期限自动计算**：
   - 基于申请日、优先权日、各法定期间自动计算
   - 考虑各国节假日、顺延规则
2. **提醒与预警**：
   - 多级提醒（到期前30天、15天、7天、1天）
   - 邮件/弹窗/桌面通知
   - 责任人 + 抄送人
3. **升级机制**：
   - 到期未完成自动升级到上级
   - 升级规则可配置
4. **期限日历**：
   - 日历视图展示所有期限
   - 按责任人、按类型筛选
5. **完成确认**：
   - 标记完成、上传证明文件
   - 记录完成时间和操作人

---

### 2.5 保密与数据安全

**之前的盲区**：
提到了数据安全，但没有充分认识到专利数据的敏感度。

**为什么重要**：
- 未公开的专利申请 = 商业秘密/国家秘密
- 泄露可能导致专利丧失新颖性
- 竞争对手得到会造成重大损失

**安全要求**：
1. **存储安全**：
   - 数据库文件加密（AES-256）
   - 备份文件加密
   - 附件加密存储
2. **访问安全**：
   - 启动密码/指纹/Windows凭据
   - 自动锁定（一段时间无操作）
   - 登录失败锁定
3. **操作审计**：
   - 所有操作记录日志（谁、什么时候、做了什么）
   - 数据变更历史（改了什么、旧值新值）
   - 导出记录（导出了什么、导出到哪）
4. **防泄露**：
   - 导出文件加水印
   - 敏感字段显示脱敏
   - 禁止复制（可选）
5. **网络安全**：
   - 本地优先，核心功能不依赖网络
   - AI功能默认本地模型，云端可选
   - 网络请求可审计

---

### 2.6 费用与成本管理

**之前的盲区**：
完全没提费用管理。

**费用类型**：

| 类别 | 明细 | 重要性 |
|------|------|--------|
| **官费** | 申请费、实审费、年费、印花税、复审费、无效费... | ⭐⭐⭐⭐⭐ |
| **代理费** | 申请代理费、答复代理费、检索代理费、无效代理费... | ⭐⭐⭐⭐ |
| **翻译费** | 说明书翻译、权利要求翻译、审查意见翻译... | ⭐⭐⭐ |
| **其他费用** | 检索费、公证费、认证费、差旅费... | ⭐⭐ |

**管理需求**：
- 费用预估（申请前估算总费用）
- 实际费用记录与追踪
- 年费预测（未来N年的年费支出）
- 按部门/项目/产品线统计
- 预算管理
- 费用缴纳提醒

---

### 2.7 专利质量与分级

**之前的盲区**：
默认所有专利都是平等的，但实际上专利价值天差地别。

**分级维度**：

| 维度 | 说明 |
|------|------|
| **技术价值** | 技术先进性、创新高度、技术难度 |
| **商业价值** | 市场规模、竞品情况、产品相关性、收益预期 |
| **法律价值** | 权利要求稳定性、保护范围宽窄、可规避性 |
| **战略价值** | 布局意义、威慑作用、交叉许可筹码 |

**分级方式**：
- 人工评定 + AI辅助评分
- 多级分类（如S/A/B/C/D五级）
- 分级标签 + 分级管理策略
- 不同等级的维护成本控制策略

## 三、其他需要注意的方面

### 3.1 部署运维类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **一键安装/卸载** | 双击安装包，下一步到底，不能让用户装运行时 | P0 |
| **绿色版可选** | 解压即用，不用安装，适合U盘携带 | P1 |
| **数据自动备份** | 定时自动备份，保留多个历史版本 | P0 |
| **手动备份/恢复** | 用户可以随时导出备份文件、从备份恢复 | P0 |
| **版本升级** | 平滑升级，数据自动迁移，可回滚 | P1 |
| **离线可用** | 核心功能（表格、查询、编辑）完全离线 | P0 |
| **多人间共享** | 数据如何在团队间共享？（共享文件/中心服务器/P2P） | P2 |
| **系统托盘** | 最小化到托盘，后台运行，期限提醒 | P1 |
| **开机自启** | 可选，用户自己决定 | P2 |

### 3.2 数据质量类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **导入数据校验** | 导入前检查格式、类型、必填项，给出错误报告 | P0 |
| **去重合并可视化** | 合并前预览差异，用户确认哪些字段用哪个版本 | P0 |
| **字段变更历史** | 每个字段的每次变更都有记录，可追溯 | P1 |
| **数据完整性校验** | 定期检查数据一致性（比如引用的专利不存在了） | P2 |
| **批量操作** | 批量编辑、批量打标签、批量改状态、批量删除 | P1 |
| **撤销/重做** | 操作失误可以撤销（至少最近几步） | P2 |

### 3.3 用户体验类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **Excel操作习惯** | 快捷键、填充柄、筛选、排序、复制粘贴尽量和Excel一致 | P0 |
| **右键菜单** | 表格、节点、标签都有右键菜单，常用操作一键直达 | P1 |
| **拖拽操作** | 拖拽排序、拖拽打标签、拖拽分组 | P2 |
| **全局快捷键** | 常用操作有快捷键（Ctrl+F搜索、Ctrl+N新建等） | P2 |
| **暗色模式** | 长时间使用的用户需要 | P3 |
| **自定义布局** | 用户可以调整侧边栏、列宽、视图布局并保存 | P2 |

### 3.4 扩展性类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **插件系统** | 功能扩展通过插件实现，不碰核心代码 | P2 |
| **REST API** | 提供完整的API，方便和其他系统集成 | P1 |
| **Webhook** | 事件通知（专利状态变更、期限到期等） | P2 |
| **自定义脚本** | 高级用户可以写脚本扩展功能 | P3 |
| **自定义仪表盘** | 用户可以自己搭看板 | P2 |

### 3.5 国际化类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **中英文双语** | 界面中英文切换，跨国团队需要 | P1 |
| **多语言数据** | 专利标题/摘要可能有多语言版本 | P1 |
| **日期格式本地化** | 不同国家的日期显示习惯不同 | P2 |
| **时区处理** | 跨国团队的期限计算要考虑时区 | P2 |

### 3.6 性能类

| 事项 | 说明 | 优先级 |
|------|------|--------|
| **冷启动速度** | 目标：3秒内进入主界面 | P1 |
| **大表滚动流畅** | 万行数据滚动不卡（虚拟滚动） | P0 |
| **导入速度** | 1万行数据导入 < 30秒 | P1 |
| **查询响应** | 普通筛选 < 1秒，复杂查询 < 3秒 | P1 |
| **图谱渲染** | 1000节点 + 3000边流畅交互 | P2 |

## 四、风险应对总原则

1. **MVP只做最确定的部分**：不确定的功能后放，先把核心做扎实
2. **配置化优于硬编码**：业务规则尽量做成可配置的，降低业务理解偏差的风险
3. **渐进式验证**：每做一个模块都找真实用户验证，不要闭门造车
4. **做好回滚预案**：数据迁移、版本升级都要有回滚方案
5. **性能前置考虑**：不要等性能出问题再优化，架构设计时就考虑性能
