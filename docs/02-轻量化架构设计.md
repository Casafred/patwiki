# 轻量化架构设计方案

> 核心原则：开箱即用、双击运行、无需安装环境、同事电脑直接能用
> 更新日期：2026-07-19

## 一、架构选型决策

### 1.1 核心约束

| 约束项 | 要求 | 严格程度 |
|--------|------|---------|
| **安装方式** | 双击安装包 / 解压即用 | 必须 |
| **Python环境** | 用户不需要安装Python | 必须 |
| **其他依赖** | 不需要装数据库、不需要装运行时 | 必须 |
| **目标用户** | 非技术人员（专利工程师、研发、产品） | 必须 |
| **运行环境** | Windows为主，Mac/Linux可选 | 高 |
| **数据安全** | 数据本地存储，不上传云端 | 必须 |
| **离线可用** | 核心功能完全离线 | 高 |

### 1.2 可选方案对比

| 方案 | 技术栈 | 体积 | 启动速度 | AI能力 | 开发效率 | 推荐度 |
|------|--------|------|---------|--------|---------|--------|
| **方案A：PyInstaller + Web前端** | Python后端 + React前端 + SQLite | 200-400MB | 3-8s | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **方案B：Tauri + Rust后端** | Rust后端 + React前端 + SQLite | 30-80MB | 1-3s | ⭐⭐ | ⭐⭐ | ⭐⭐⭐ |
| **方案C：Electron + Node.js** | Node.js后端 + React前端 + SQLite | 100-200MB | 2-5s | ⭐⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐ |
| **方案D：纯前端WASM** | 纯前端 + IndexedDB + WASM | 20-50MB | <1s | ⭐ | ⭐ | ⭐⭐ |

### 1.3 推荐方案：方案A（PyInstaller + Web前端）

**核心理由**：
1. **AI能力最强**：Python是AI生态的绝对中心，向量检索、NLP、模型推理都最成熟
2. **开发效率最高**：后端Python、前端React，都是成熟生态
3. **数据处理能力强**：pandas、openpyxl等数据处理库Python最强
4. **打包成熟**：PyInstaller虽然体积大，但很成熟，企业级应用常用

**体积优化策略**：
- 懒加载：AI相关库按需导入，启动时不加载
- 模块裁剪：只打包用到的功能，去掉测试、文档等
- UPX压缩：可执行文件压缩
- 分发包拆分：核心包 + AI扩展包（可选下载）

**启动速度优化策略**：
- 启动时只加载核心模块（数据库、API、静态文件服务）
- AI、向量检索等重型模块首次使用时再加载
- 显示启动进度条，降低等待焦虑
- 系统托盘后台运行，下次秒开

| 层级 | 技术选型 | 选型理由 |

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| **Web框架** | FastAPI | 性能好、自动文档、类型安全、Python生态 |
| **ORM** | SQLAlchemy 2.0 | Python最成熟的ORM，功能强大 |

| 层级 | 技术选型 | 选型理由 |

## 四、数据存储设计

### 4.1 存储结构

```
用户数据目录（默认：C:\Users\xxx\Documents\PatWiki\）
├── data\
│   └── patwiki.db          # SQLite主数据库文件
├── vectors\
│   └── patent_index\       # Zvec向量索引目录
├── files\
│   ├── patents\            # 专利附件（PDF、图片等）
│   ├── documents\          # 其他文档
│   └── avatars\            # 头像
├── backups\                # 自动备份目录
│   └── 2026-07-19.db
├── cache\                  # 缓存目录
├── logs\                   # 日志目录
└── config.json             # 应用配置
```

### 4.2 为什么选SQLite而不是PostgreSQL？

| 对比项 | SQLite | PostgreSQL |
|--------|--------|------------|
| **安装** | 不需要，就是一个文件 | 需要安装服务 |
| **配置** | 零配置 | 需要配置端口、用户、权限 |
| **体积** | 几百KB | 几百MB |
| **启动** | 随应用启动 | 需要单独启动服务 |
| **并发** | 读写有锁，并发一般 | 高并发 |
| **功能** | JSONB、全文搜索都支持 | 功能最强 |
| **备份** | 拷贝文件就行 | 需要pg_dump |
| **适合场景** | 单用户/小团队 | 多用户、大数据量 |

**结论**：本地单用户场景，SQLite完全够用，而且轻量太多。真要多用户了再考虑升级。

### 4.3 向量存储方案

- **Zvec**：嵌入式向量库，和Python后端一起打包，不需要额外服务
- 向量索引文件存在 `vectors/` 目录下
- 可以随时重建（从数据库重新向量化）
- 支持混合检索（向量相似度 + 标量过滤）

## 五、应用启动流程

```
用户双击 exe
    ↓
显示启动画面（进度条）
    ↓
加载核心模块（配置、日志、数据库连接）
    ↓
启动 FastAPI 服务（本地端口，只监听127.0.0.1）
    ↓
打开 WebView 窗口
    ↓
前端加载完成，显示主界面
    ↓
（首次使用时）初始化数据库、创建默认数据
```

**性能目标**：
- 冷启动：5秒内进入主界面
- 热启动（托盘恢复）：<1秒
- 数据库查询：普通查询 < 100ms，复杂查询 < 1s

## 六、模块拆分与懒加载策略

### 6.1 模块分级

| 级别 | 模块 | 加载时机 | 说明 |
|------|------|---------|------|
| **核心模块** | 配置、日志、数据库、用户认证 | 启动时 | 必须加载 |
| **基础模块** | 专利CRUD、表格、标签、项目 | 启动时 | 核心功能 |
| **按需加载** | AI模块、向量检索、图谱 | 首次使用时 | 重型模块，懒加载 |
| **可选模块** | 高级分析、报表、工作流 | 安装时可选 | 用户按需安装 |

### 6.2 懒加载实现

**后端懒加载**：
- 重型模块（AI、向量）用工厂模式，首次调用时初始化
- 导入语句放在函数内部，而不是文件顶部
- 使用单例模式，确保只初始化一次

**前端懒加载**：
- 基于路由的代码分割（React.lazy）
- 大型组件（图谱、富文本编辑器）按需加载
- 骨架屏 + 加载动画，提升体验

## 七、打包与分发方案

### 7.1 PyInstaller 打包策略

**打包方式**：单目录模式（onedir），不是单文件模式（onefile）
- 单目录模式启动更快（不需要解压到临时目录）
- 更新时可以只替换部分文件
- 调试更方便

**体积优化**：
```
1. 排除不需要的模块
   --exclude-module tkinter
   --exclude-module matplotlib
   --exclude-module scipy (如果不用的话)
   ...

2. UPX 压缩可执行文件

3. 只导入需要的函数，而不是整个包

4. 数据集、模型文件单独下载，不打进安装包
```

### 7.2 安装包（Windows）

**使用 Inno Setup**：
- 标准Windows安装体验
- 可选安装路径
- 创建桌面快捷方式、开始菜单
- 可选开机自启
- 卸载程序
- 文件关联（.patwikibackup 备份文件）

### 7.3 绿色版（可选）

- 解压即用，不需要安装
- 适合U盘携带
- 所有数据和程序在同一个目录
- 不会写入系统注册表

## 九、数据安全设计

### 9.1 存储加密

- **数据库加密**：SQLite加密扩展（SQLCipher），整个数据库文件加密
- **向量索引加密**：Zvec数据文件加密存储
- **文件加密**：附件文件AES加密存储
- **加密密钥**：用户密码派生（PBKDF2），或Windows凭据存储

### 9.2 访问控制

- **启动密码**：应用启动需要密码
- **自动锁定**：一段时间无操作自动锁定
- **失败锁定**：连续输错密码多次，锁定一段时间

### 9.3 网络安全

- **后端只监听127.0.0.1**：不暴露到外网
- **API认证**：本地API也需要token认证
- **网络请求审计**：所有外网请求都有日志
- **云端AI可选**：用户主动开启才调用云端API，默认本地/离线

## 十、升级与迁移策略

### 10.1 版本升级

- **检查更新**：启动时可选检查更新
- **增量更新**：只下载变化的文件
- **自动备份**：升级前自动备份数据库
- **一键回滚**：升级失败可以回滚到旧版本

### 10.2 数据迁移

- 使用 Alembic 管理数据库版本
- 每个版本有升级脚本和降级脚本
- 升级前自动备份，升级失败自动恢复
- 大版本升级提供迁移工具

## 十一、备选方案：如果Python体积实在太大

### 备选方案：Tauri + Rust后端

**适用场景**：体积和启动速度要求极高，AI需求不强

**优点**：
- 体积极小（30-80MB）
- 启动极快（1-3s）
- 内存占用低
- 安全性好（Rust + 沙箱）

**缺点**：
- Rust AI生态弱，向量检索可用，但复杂AI功能受限
- 开发效率比Python低
- 数据处理库不如Python丰富

**折衷方案**：
- 核心用Tauri + Rust
- AI功能通过调用本地Python子进程实现（Python运行时打包在内部）
- 或者AI功能纯云端调用
