## 二、代码组织基本原则

### 2.1 文件大小限制

| 文件类型 | 建议最大行数 | 超过时处理方式 |
|---------|-------------|--------------|
| Python 模块 | 300 行 | 拆分为多个子模块 |
| Python 类 | 200 行 | 拆分职责，使用组合 |
| Python 函数 | 50 行 | 拆分为多个小函数 |
| React 组件 | 300 行 | 拆分为子组件 + Hooks |
| TypeScript 类型文件 | 200 行 | 按领域拆分 |

**原则**：
- 一个文件/类/函数只做一件事（单一职责原则）
- 当文件超过限制时，优先考虑拆分，而不是继续往里加
- 宁可多几个小文件，不要一个大文件

### 2.2 分层原则

```
表现层 (前端React)
    ↓
API 层 (FastAPI Routes)
    ↓
服务层 (Services)  ←── 业务逻辑核心
    ↓
模型层 (SQLAlchemy Models)
    ↓
基础设施层 (数据库、存储、外部API)
```

**调用规则**：
- 上层可以调用下层，下层不能调用上层
- 同层之间尽量减少互相调用
- 业务逻辑写在 Service 层，不写在 API 层
- API 层只做参数校验、调用 Service、返回响应

### 2.3 前端组件分层原则

```
页面级组件 (Page)
    ↓
特性级组件 (Feature)
    ↓
业务组件 (Business Component)
    ↓
通用组件 (Common / UI Component)
```

**规则**：
- `components/ui/` 是纯UI组件，不包含任何业务逻辑
- `features/xxx/components/` 是模块内可复用的业务组件
- 页面组件只做组装，不包含复杂逻辑
- 复杂逻辑抽成自定义 Hook

## 三、命名规范

### 3.1 Python 后端

| 类型 | 命名规范 | 示例 |
|------|---------|------|
| **模块/文件** | 蛇形命名（snake_case） | `patent_service.py`, `import_service.py` |
| **类** | 大驼峰（PascalCase） | `PatentService`, `AIFieldEngine` |
| **函数/方法** | 蛇形命名 | `get_patent_by_id()`, `calculate_ai_field()` |
| **变量** | 蛇形命名 | `patent_list`, `custom_fields` |
| **常量** | 全大写 + 下划线 | `MAX_RETRY_COUNT`, `DEFAULT_PAGE_SIZE` |
| **私有成员** | 下划线前缀 | `_internal_method()`, `_cached_value` |

### 3.2 TypeScript 前端

| 类型 | 命名规范 | 示例 |
|------|---------|------|
| **文件（组件）** | 大驼峰（PascalCase） | `PatentTable.tsx`, `TagSelector.tsx` |
| **文件（工具/类型）** | 小驼峰（camelCase） | `utils.ts`, `patentTypes.ts` |
| **组件** | 大驼峰 | `PatentDetailPage`, `AIFieldEditor` |
| **函数/Hook** | 小驼峰 | `getPatentList()`, `useDebounce()` |
| **变量** | 小驼峰 | `patentList`, `customFields` |
| **常量** | 全大写 + 下划线 | `MAX_PAGE_SIZE`, `API_BASE_URL` |
| **类型/接口** | 大驼峰，I前缀可选 | `Patent`, `FieldConfig`, `IPatentService` |
| **CSS类名** | kebab-case | `patent-card`, `field-sidebar` |

### 3.3 数据库命名

| 类型 | 命名规范 | 示例 |
|------|---------|------|
| **表名** | 蛇形 + 复数 | `patents`, `custom_fields`, `patent_tags` |
| **列名** | 蛇形 | `patent_id`, `application_number`, `filing_date` |
| **主键** | `id` | `id` |
| **外键** | 表名单数 + `_id` | `patent_id`, `project_id` |
| **时间戳** | `created_at`, `updated_at` | `created_at`, `updated_at` |
| **索引** | `idx_` + 表名 + 列名 | `idx_patents_application_number` |

## 四、模块开发规范

### 4.1 新增业务模块的步骤

新增一个业务模块（如"费用管理"），按以下顺序创建：

1. **Model 层**：`models/fee/` - 数据模型
2. **Schema 层**：`schemas/fee/` - 请求/响应数据结构
3. **Service 层**：`services/fee_service.py` - 业务逻辑
4. **API 层**：`api/fees.py` - API 路由
5. **前端类型**：`frontend/src/types/fee.ts` - TypeScript 类型
6. **前端 API**：`frontend/src/api/fee.ts` - API 调用
7. **前端组件**：`frontend/src/features/fee/` - 业务组件
8. **前端路由**：在 `router.tsx` 中注册路由

### 4.2 模块间依赖规则

- **允许的依赖**：
  - 通用模块 → 被所有模块依赖（utils, config, core）
  - 业务模块 → 可以依赖通用模块
  - 业务模块 → 可以依赖其他业务模块的 Service 层
- **禁止的依赖**：
  - 通用模块 → 依赖业务模块
  - API 层 → 直接调用其他 API 层（应该调用 Service 层）
  - Model 层 → 依赖 Service 层

## 五、代码质量规范

### 5.1 Python 代码规范

- 遵循 PEP 8
- 使用 Black 格式化代码
- 使用 isort 整理 import
- 使用 ruff 做 lint
- 类型注解：所有公共函数必须有类型注解

```python
# 好的示例
def get_patent(patent_id: str) -> Patent | None:
    """根据专利ID获取专利信息"""
    return db.query(Patent).filter(Patent.id == patent_id).first()

# 不好的示例
def get_patent(pid):
    # 没有类型注解，没有文档
    return db.query(Patent).filter(Patent.id == pid).first()
```

### 5.2 TypeScript 代码规范

- 严格模式（strict: true）
- 禁止使用 `any`（除非万不得已，且有注释说明）
- 组件用函数组件 + Hooks
- Props 必须定义类型

```typescript
// 好的示例
interface PatentCardProps {
  patent: Patent;
  onClick?: (patent: Patent) => void;
}

const PatentCard: React.FC<PatentCardProps> = ({ patent, onClick }) => {
  return <div onClick={() => onClick?.(patent)}>{patent.title}</div>;
};

// 不好的示例
const PatentCard = ({ patent, onClick }: any) => {
  // 用了 any，类型不安全
  return <div onClick={onClick}>{patent.title}</div>;
};
```

### 5.3 注释规范

- **不要写"代码做了什么"的注释**（代码本身应该能说明）
- **写"为什么这么做"的注释**（设计决策、业务背景、注意事项）
- 公共 API 必须有文档字符串/TSDoc
- 复杂算法必须有注释说明思路

```python
# 好的示例
# 为什么用二分查找：因为列表已排序，O(logn)比O(n)快很多
def binary_search(sorted_list: list[int], target: int) -> int:
    """在已排序的列表中二分查找目标值"""
    ...

# 不好的示例
# 循环找目标
def find_target(lst, target):
    # 遍历列表
    for item in lst:
        # 如果找到了
        if item == target:
            # 返回True
            return True
    # 没找到返回False
    return False
```
