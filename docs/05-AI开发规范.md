# AI 开发基础规范

> 核心原则：可预期、可追溯、可控制、用户可理解
> 更新日期：2026-07-19

## 一、AI功能设计总原则

### 1.1 三大铁律

| 原则 | 说明 | 违反后果 |
|------|------|---------|
| **可预期** | 用户知道AI什么时候在工作、输出什么类型的结果 | 用户困惑、不信任 |
| **可追溯** | 每个AI输出都能追溯：用了什么模型、什么提示词、什么输入 | 出问题找不到原因 |
| **可控制** | 用户可以干预、修改、关闭AI功能 | 用户感觉失控 |

### 1.2 用户体验原则

1. **AI是助手，不是主角**：
   - AI输出永远是"建议"，不是"决定"
   - 最终决定权在用户手里
   - 人工修改优先于AI自动更新

2. **明确标识AI内容**：
   - AI生成的内容要有明确的"AI生成"标识
   - AI字段和普通字段视觉上有区分
   - 让用户清楚哪些是AI算的，哪些是人写的

3. **失败优雅降级**：
   - AI调用失败不能导致整个功能崩溃
   - 有友好的错误提示和重试机制
   - 离线状态下基础功能正常

4. **成本可控**：
   - AI调用要有成本意识，不能随便调用
   - 批量操作前提示预估成本
   - 用户可以设置每日/每月调用上限

## 二、AI功能模块划分

### 2.1 AI功能全景

```
AI 能力层
├── AI 字段
│   ├── 文本生成类（摘要、技术要点提取、翻译）
│   ├── 分类类（技术领域分类、风险等级评定）
│   ├── 评分类（创新性评分、价值评分）
│   └── 提取类（从文本中提取结构化信息）
│
├── AI 标签
│   ├── 自动打标签
│   ├── 标签推荐
│   └── 标签传播（相似专利传播标签）
│
├── 自然语言查询
│   ├── NL → 筛选条件
│   ├── NL → 统计/聚合
│   └── NL → 生成新视图
│
├── 智能检索
│   ├── 向量语义搜索
│   ├── 混合检索（关键词 + 向量）
│   └── 相似专利推荐
│
├── 智能导入
│   ├── 智能字段映射（自动识别Excel列对应哪个字段）
│   ├── 智能去重（标题相似度匹配）
│   └── 智能合并（冲突字段的合并建议）
│
├── 内容生成
│   ├── 专利摘要生成
│   ├── 技术分析报告生成
│   ├── 审查意见答复建议（远期）
│   └── 风险评估报告生成
│
└── AI 工作流（远期）
    ├── 批量AI处理
    ├── 条件触发AI动作
    └── AI + 人工审批流程
```

## 三、AI字段设计规范

### 3.1 AI字段定义

```python
AIFieldDefinition:
    # 基本信息
    name: str                    # 字段名称
    description: str             # 字段描述
    field_type: str              # 输出类型（text/number/select/multi_select/date）
    
    # AI配置
    prompt_template: str         # 提示词模板
    model: str                   # 使用的模型
    temperature: float = 0.0     # 温度参数（默认0，保证确定性）
    
    # 触发方式
    trigger_mode: str            # auto（自动）/ manual（手动）
    auto_trigger:                # 自动触发条件
        - on_create              # 专利创建时
        - on_update              # 专利更新时
        - schedule               # 定时触发
    
    # 输出配置
    output_schema: dict          # 输出的JSON Schema（结构化输出用）
    enum_options: list[str]      # 如果是select类型，选项列表
    
    # 缓存配置
    cache_enabled: bool = True   # 是否启用缓存
    cache_key_fields: list[str]  # 哪些字段变化时失效缓存
```

### 3.2 AI字段的状态

| 状态 | 说明 | UI展示 |
|------|------|--------|
| `pending` | 等待计算 | 显示"计算中..." |
| `success` | 计算成功 | 正常显示，带AI标识 |
| `failed` | 计算失败 | 显示失败，可重试 |
| `human_overridden` | 人工覆盖 | 显示值，带"人工确认"标识 |
| `outdated` | 已过期（输入变了） | 显示旧值，有"已过期"提示，可重新计算 |

### 3.3 AI字段版本管理

每次AI计算都要记录：
- 计算时间
- 模型名称和版本
- 提示词版本
- 温度等参数
- 输入数据的哈希（用于判断输入是否变化）
- 输出结果
- 耗时
- 调用成本（token数、费用）

**为什么？**
- 用户问"为什么这个值变了？"时能回答
- 模型升级后，可以对比新旧结果
- 可以回放和复现

## 四、提示词工程规范

### 4.1 提示词模板结构

```python
prompt_template = """
# 角色定义
你是一个专利领域专家，擅长{task_description}。

# 任务说明
请根据提供的专利信息，{specific_task}。

# 输出要求
- 输出格式：{output_format_description}
- 如果是结构化输出，严格按照JSON格式
- {additional_constraints}

# 专利信息
标题：{patent.title}
摘要：{patent.abstract}
权利要求：{patent.claims}
技术领域：{patent.technical_field}

{custom_fields_section}

# 输出
"""
```

### 4.2 提示词版本管理

- 每个提示词模板有版本号
- 修改提示词创建新版本，不覆盖旧版本
- AI字段可以选择使用哪个版本的提示词
- 可以对比不同版本提示词的输出效果

### 4.3 结构化输出规范

**优先使用结构化输出（JSON）**，原因：
- 容易解析和验证
- 可以直接填充到字段中
- 可以做类型检查

**实现方式**：
- 使用 JSON Schema 约束输出格式
- 使用函数调用（Function Calling）/ 工具调用
- 输出后做 JSON 校验和类型转换
- 校验失败自动重试（最多3次）

### 4.4 温度参数默认值

| 场景 | 推荐温度 | 理由 |
|------|---------|------|
| 分类/评分 | 0.0 | 需要确定性结果 |
| 提取信息 | 0.0 | 需要准确提取 |
| 翻译 | 0.0 | 需要准确 |
| 摘要 | 0.1-0.3 | 稍微灵活，但不要太发散 |
| 创意/头脑风暴 | 0.7-1.0 | 需要多样性 |

**默认值：0.0**——AI字段是"字段"，应该尽量确定。

## 五、模型供应商适配规范

### 5.1 模型抽象层

```python
class BaseLLMProvider(ABC):
    """LLM供应商基类，所有供应商必须实现这些接口"""
    
    @abstractmethod
    def chat_completion(self, messages: list, **kwargs) -> str:
        """聊天补全"""
        pass
    
    @abstractmethod
    def structured_output(self, messages: list, schema: dict, **kwargs) -> dict:
        """结构化输出（JSON）"""
        pass
    
    @abstractmethod
    def embedding(self, texts: list[str]) -> list[list[float]]:
        """文本向量化"""
        pass
    
    @abstractmethod
    def count_tokens(self, text: str) -> int:
        """计算token数"""
        pass
```

### 5.2 内置供应商

| 供应商 | 适用场景 | 优先级 |
|--------|---------|--------|
| **智谱AI (Zhipu)** | 国内用户，中文好，速度快 | P0 |
| **OpenAI** | 效果最好，功能最全 | P1 |
| **通义千问 (Qwen)** | 阿里系，国内可用 | P1 |
| **文心一言 (ERNIE)** | 百度，国内可用 | P2 |
| **本地模型 (Ollama)** | 完全离线，数据不出本地 | P2 |

### 5.3 模型配置

- 用户可以在设置中配置自己的API Key
- 可以选择默认模型
- 每个AI字段可以单独指定模型
- 支持降级策略（主模型失败自动用备用模型）

## 六、向量检索设计规范

### 6.1 向量存储架构

```
向量检索层
├── VectorStore（抽象接口）
│   ├── ZvecVectorStore（Zvec实现，默认）
│   ├── PGVectorStore（pgvector实现，备选）
│   └── ChromaVectorStore（Chroma实现，备选）
├── EmbeddingService（向量化服务）
│   ├── 批量向量化
│   ├── 增量更新
│   └── 重建索引
└── SearchService（检索服务）
    ├── 纯向量检索
    ├── 混合检索（向量 + 关键词）
    └── 带过滤的检索（向量 + 属性过滤）
```

### 6.2 专利文本向量化策略

**不要把整个专利直接向量化**——内容太长，信息密度低。

**分块策略**：

| 分块方式 | 内容 | 适用场景 |
|---------|------|---------|
| **摘要块** | 专利摘要 | 快速语义搜索 |
| **权利要求块** | 每条权利要求单独向量化 | 精确的权利要求相似性 |
| **技术领域块** | 技术领域 + 背景技术 | 技术领域分类 |
| **实施例块** | 每个实施例单独向量化 | 具体技术方案相似性 |
| **全文块** | 说明书按段落分块 | 全文语义搜索 |

**元数据**：
每个向量都要带元数据，用于过滤：
- patent_id: 专利ID
- chunk_type: 分块类型（abstract/claim/description...）
- chunk_index: 分块序号
- technical_field: 技术领域
- filing_date: 申请日
- status: 法律状态

### 6.3 混合检索

```
用户查询
    ↓
┌─────────────────┐    ┌─────────────────┐
│  关键词检索      │    │  向量检索       │
│  (BM25/TF-IDF)  │    │  (相似度)      │
└────────┬────────┘    └────────┬────────┘
         │                       │
         └───────────┬───────────┘
                     ↓
              结果融合（RRF/加权）
                     ↓
               重排序（可选）
                     ↓
                 最终结果
```

### 6.4 向量索引维护

- **增量更新**：专利新增/修改时，自动更新向量
- **批量重建**：可以全量重建向量索引（换模型、改分块策略时）
- **索引状态**：显示索引健康度、已索引数量、待处理数量

## 七、缓存与成本控制规范

### 7.1 缓存策略

| 缓存级别 | 缓存内容 | 缓存键 | 有效期 |
|---------|---------|--------|--------|
| **字段级** | AI字段计算结果 | 专利ID + 字段ID + 输入哈希 + 模型版本 | 输入不变就有效 |
| **Embedding级** | 文本向量 | 文本内容哈希 + Embedding模型 | 永久（除非换模型） |
| **查询级** | NL查询的解析结果 | 查询文本 + 模型版本 | 7天 |

### 7.2 成本控制

1. **调用预算**：
   - 用户可以设置每日/每月token上限
   - 快到上限时提醒
   - 超出后暂停自动AI功能

2. **批量操作确认**：
   - 批量处理超过N条时，弹出确认框
   - 显示预估token数和费用
   - 用户确认后再执行

3. **懒计算**：
   - AI字段默认不自动计算
   - 用户查看时才计算（lazy evaluation）
   - 或者后台低优先级队列慢慢算

4. **复用优先**：
   - 相同输入 + 相同模型 + 相同提示词 = 直接返回缓存
   - 不要重复计算

## 八、错误处理与降级规范

### 8.1 常见错误类型

| 错误类型 | 原因 | 处理方式 |
|---------|------|---------|
| **网络错误** | 断网、API连不上 | 显示网络错误，提供重试按钮 |
| **API Key无效** | Key错了、过期了 | 提示用户检查配置 |
| **额度不足** | 余额不够了 | 提示充值/换Key |
| **限流** | 调用太频繁 | 自动退避重试，提示用户稍后再试 |
| **输出解析失败** | AI返回格式不对 | 自动重试（最多3次），再不行显示原始输出 |
| **超时** | 响应太慢 | 后台继续执行，通知用户完成 |
| **内容审核不通过** | 输出被过滤 | 友好提示，不显示敏感内容 |

### 8.2 降级策略

1. **功能降级**：
   - AI功能不可用时，不影响核心功能（表格、编辑、导入导出）
   - AI字段显示"AI功能暂不可用"，但不影响其他字段

2. **模型降级**：
   - 主模型失败，自动尝试备用模型
   - 大模型失败，尝试小模型

3. **精度降级**：
   - 详细模式失败，尝试简单模式
   - 结构化输出失败，退化为纯文本输出

### 8.3 用户可见的错误提示原则

- **不说技术术语**：不说"API返回429"，说"调用太频繁了，稍后再试"
- **给出解决方案**：告诉用户可以做什么（重试、检查配置、稍后再试）
- **不暴露敏感信息**：不把API Key、原始错误堆栈显示给用户
- **记录详细日志**：用户看不到，但开发人员能调试

## 九、AI功能开发流程规范

### 9.1 新增AI功能的步骤

1. **需求定义**：
   - 明确AI功能要解决什么问题
   - 明确输入输出
   - 明确成功标准

2. **提示词设计与测试**：
   - 设计初版提示词
   - 用20+个测试用例验证效果
   - 迭代优化提示词

3. **结构化输出设计**：
   - 设计输出的JSON Schema
   - 验证输出的可解析性
   - 设计容错机制

4. **集成开发**：
   - 实现后端逻辑
   - 实现前端UI
   - 接入缓存和错误处理

5. **测试验证**：
   - 功能测试
   - 边界测试（空输入、超长输入...）
   - 错误注入测试（网络断了、API挂了...）
   - 性能测试（响应时间、并发）

6. **灰度上线**：
   - 先给少数用户用
   - 收集反馈
   - 迭代优化

### 9.2 测试用例要求

每个AI功能至少要有：
- 10个正常用例（典型场景）
- 5个边界用例（超长文本、空内容、模糊描述...）
- 3个错误用例（网络错、API错、输出格式错...）
- 准确率/召回率评估（如果是分类/提取类）

## 十、安全与隐私规范

### 10.1 数据安全

1. **本地优先**：
   - 默认数据本地处理
   - 云端AI是可选功能，用户主动开启

2. **数据最小化**：
   - 只发送必要的数据给AI
   - 敏感字段（如未公开的发明内容）默认不发送
   - 用户可以配置哪些字段可以发送给AI

3. **发送前确认**：
   - 首次使用云端AI时，明确提示数据会发送到第三方
   - 用户确认后才能使用
   - 可以随时关闭

4. **审计日志**：
   - 记录每次AI调用：发送了什么、收到了什么、什么时候、用了多少token
   - 用户可以查看和导出

### 10.2 内容安全

1. **输出审核**：
   - 敏感内容过滤
   - 不当内容拒绝生成

2. **用户反馈**：
   - AI输出有问题，用户可以"踩"或举报
   - 收集bad case用于优化

## 十一、可观测性规范

### 11.1 需要监控的指标

| 指标 | 说明 | 目的 |
|------|------|------|
| **调用次数** | 每天/每周调用多少次 | 用量统计 |
| **成功率** | 成功调用占比 | 稳定性监控 |
| **平均响应时间** | 调用耗时 | 性能监控 |
| **Token用量** | 消耗了多少token | 成本监控 |
| **缓存命中率** | 多少请求命中了缓存 | 优化效果 |
| **用户满意度** | 点赞/点踩比例 | 质量监控 |
| **错误分布** | 各种错误类型的比例 | 问题定位 |

### 11.2 日志规范

每次AI调用都要记录：
- 时间戳
- 功能类型（AI字段/标签/NL查询...）
- 模型供应商和模型名
- 输入（或输入的哈希/摘要）
- 输出
- 耗时
- token数
- 是否成功
- 错误信息（如果失败）
- 是否命中缓存

**注意**：日志要考虑隐私，敏感数据可以脱敏或只存哈希。

## 十二、MVP阶段的AI功能范围

### 12.1 MVP 做什么

**轻量起步，验证价值**：

1. **AI字段（基础版）**
   - 文本生成类：摘要生成、技术要点提取
   - 分类类：技术领域分类（预设几个大类）
   - 手动触发，不自动计算
   - 简单的提示词配置

2. **向量语义搜索**
   - 专利摘要向量化
   - 简单的语义搜索
   - Zvec存储
   - 智谱/OpenAI Embedding

3. **智能打标签（基础版）**
   - 基于关键词规则的标签推荐（先不用AI）
   - 或者简单的AI标签推荐

### 12.2 MVP 不做什么

- 自然语言查询（第二期）
- 智能导入（第二期）
- AI工作流（第三期）
- 本地模型（第二期以后）
- 多模型对比（第二期）
- 成本统计面板（第二期）
