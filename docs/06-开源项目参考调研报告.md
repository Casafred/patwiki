# 专利多维数据库管理系统 - 开源项目参考调研报告

> 调研日期：2026-07-19
> 调研范围：多维表格、知识管理/Wiki、图谱可视化、向量数据库、专利工具、AI框架、数据处理等7大方向

---

## 执行摘要

本报告系统性调研了 **7大方向、25+个开源项目**，分析各项目的架构设计、核心价值、可借鉴点及局限性，为构建"以专利为锚点的多维知识管理系统"提供参考。

**核心结论**：
1. **没有一个现有项目能直接满足需求**——这正是本项目的价值所在
2. **最佳策略是"核心自研 + 关键组件复用"**，而非基于某一平台深度改造
3. **可重点借鉴的项目**：NocoBase（插件化架构+数据模型设计）、Baserow（公式引擎）、Logseq（双向链接+图谱交互）、Zvec（嵌入式向量检索）、Patent2Net（专利分析方法论）
4. **可直接复用的组件**：ip-tools系列API客户端、AG Grid表格引擎、Cytoscape.js/G6图谱引擎、pandas数据处理

## 一、多维表格 / 低代码平台方向

### 1.1 NocoBase

**项目简介**
- GitHub: https://github.com/nocobase/nocobase
- 技术栈：Node.js + React + PostgreSQL
- 定位：可扩展的无代码/低代码开发平台，以"扩展性"为核心
- Stars: ~13k

**核心价值与优点**
1. **微内核+插件架构**：所有功能均以插件形式存在，核心极小，扩展能力极强
2. **数据模型驱动**：通过Collection定义数据模型，支持自定义字段类型、关联关系
3. **字段类型丰富**：支持公式字段（formula）、聚合字段（rollup）、关联字段（lookup）等
4. **前后端一体化**：插件同时扩展前端和后端，统一的开发体验
5. **v1.0已发布**：相对稳定，有商业支持

**实现方式/架构特点**
```
NocoBase 架构
├── 核心层 (Core)
│   ├── 插件管理器 (PluginManager)
│   ├── 数据模型管理器 (CollectionManager)
│   └── API 网关
├── 插件层 (Plugins)
│   ├── 数据类型插件 (formula, rollup, file, etc.)
│   ├── UI 组件插件 (表格, 表单, 看板)
│   ├── 工作流插件
│   └── 认证/权限插件
└── 存储层
    └── PostgreSQL (所有表动态创建)
```

- Collection（数据表）定义存储在数据库中，运行时动态解析
- 字段类型通过插件注册，新增字段类型无需修改核心代码
- 前端基于 React + Ant Design，使用 Schema 渲染

**可借鉴点**
1. ✅ **插件化架构设计**：微内核思想，所有功能模块化，便于持续扩展
2. ✅ **数据模型抽象层**：Collection + Field 的设计模式，支持动态字段
3. ✅ **公式字段实现思路**：formula field 插件的公式解析与计算架构
4. ✅ **前后端一体化插件机制**：一个插件包同时包含前后端代码
5. ✅ **字段类型扩展机制**：如何新增自定义字段类型（如AI字段）

**局限性**
- Node.js 技术栈，与 Python AI 生态集成需额外工作
- 通用低代码平台，专利特异性业务逻辑需大量定制
- 图谱、双向链接等知识管理能力完全缺失

### 1.2 NocoDB

**项目简介**
- GitHub: https://github.com/nocodb/nocodb
- 技术栈：Node.js + Vue + SQLite/PostgreSQL
- 定位：开源Airtable替代，将数据库转为智能表格
- Stars: ~47k（星标最高）

**核心价值与优点**
1. **成熟度高**：社区活跃，功能完善，生产可用
2. **多数据库支持**：MySQL, PostgreSQL, SQL Server, SQLite 等
3. **自动API生成**：REST + GraphQL API 自动生成
4. **视图丰富**：表格、画廊、表单、看板、日历视图
5. **集成能力强**：支持与多种第三方服务集成

**实现方式/架构特点**
- 基于现有数据库 schema 反向生成表格界面
- 元数据存储在 NC_ 前缀的系统表中
- 插件系统相对较弱（integration 方向为主）

**可借鉴点**
1. ✅ **表格交互体验**：类Excel的操作体验设计
2. ✅ **多视图切换架构**：同一数据在不同视图间的切换机制
3. ✅ **API自动生成模式**：如何基于数据模型自动生成CRUD API
4. ✅ **导入导出功能**：Excel/CSV导入导出的实现方式

**局限性**
- 插件化程度不如NocoBase，深度二开难度大
- 自定义字段类型扩展不够灵活
- 工作流、计算字段能力较弱
- 代码结构较复杂，历史包袱较重

---

### 1.3 Baserow

**项目简介**
- GitHub: https://github.com/bram2w/baserow
- 技术栈：Python (Django) + Vue + PostgreSQL
- 定位：开源无代码数据库工具
- Stars: ~3k

**核心价值与优点**
1. **Python技术栈**：Django后端，与AI生态集成更方便
2. **公式引擎强大**：Baserow Formula Engine 支持复杂公式计算
3. **字段类型丰富**：formula, rollup, lookup, link row table 等
4. **代码质量高**：架构清晰，文档完善

**实现方式/架构特点**
- 后端基于 Django REST Framework
- 公式引擎：自定义公式语法解析器，支持函数调用
- 每个表对应实际的数据库表，字段对应列

**可借鉴点**
1. ✅ **公式引擎实现**：最值得参考的部分——公式语法定义、解析器、函数注册、计算优化
2. ✅ **Python后端架构**：Django + DRF 的分层设计
3. ✅ **lookup/rollup字段实现**：跨表关联计算的实现思路
4. ✅ **字段类型注册机制**：如何灵活扩展字段类型

**局限性**
- 前端是 Vue，如选React技术栈需重写前端
- 社区规模小于NocoDB
- 插件系统不如NocoBase成熟

### 1.4 Budibase / Appsmith

**项目简介**
- Budibase: 低代码平台，侧重应用构建 + 自动化流程
- Appsmith: 内部工具构建平台，侧重仪表板 + 管理后台
- 技术栈：均为 Node.js + React

**可借鉴点**
1. ✅ **可视化页面构建器**：拖拽式页面布局的实现思路
2. ✅ **数据源集成架构**：多数据源统一连接层设计
3. ✅ **工作流引擎**：自动化流程的编排机制

**局限性**
- 不是以"数据表格"为核心，表格能力弱于NocoDB/NocoBase
- 更适合搭应用而非数据管理系统

| 项目 | 最值得借鉴 | 借鉴优先级 |
|------|-----------|-----------|
| **NocoBase** | 插件化架构、数据模型抽象、字段扩展机制 | ⭐⭐⭐⭐⭐ |

**建议**：深入研读 NocoBase 的插件架构和 Baserow 的公式引擎代码，这两块最有参考价值。

## 二、知识管理 / Wiki / 双向链接方向

### 2.1 Logseq

**项目简介**
- GitHub: https://github.com/logseq/logseq
- 技术栈：ClojureScript + React
- 定位：隐私优先的开源知识库，支持双向链接、白板、图谱
- Stars: ~32k

**核心价值与优点**
1. **双向链接机制成熟**：`[[页面名]]` 语法，自动建立反向链接
2. **块级引用**：可以引用到具体某一段内容（block reference）
3. **图谱视图**：全局关系图谱可视化，支持筛选
4. **白板功能**：可视化的知识组织方式
5. **本地优先**：数据存储在本地Markdown文件

**实现方式/架构特点**
```
Logseq 核心机制
├── 数据层
│   ├── 本地 Markdown 文件
│   ├── 数据库索引 (Datascript / SQLite)
│   └── 块 (Block) 为最小单位
├── 链接系统
│   ├── [[page]] 页面链接 → 自动建立反向链接
│   ├── ((block)) 块引用
│   └── #tag 标签链接
├── 图谱视图
│   ├── 节点 = 页面/标签
│   ├── 边 = 链接关系
│   └── 力导向布局
└── 查询语言
    └── Datalog / Advanced Query
```

**可借鉴点**
1. ✅ **双向链接实现机制**：如何解析`[[...]]`语法、建立索引、查询反向链接
2. ✅ **图谱视图交互设计**：节点筛选、缩放、拖拽、点击跳转等交互
3. ✅ **块级引用思想**：专利权利要求的段落级引用/标注是否可借鉴
4. ✅ **本地优先架构**：数据本地存储 + 索引层的设计思路
5. ✅ **查询语言设计**：如何让用户用简单语法查询复杂关系

**局限性**
- ClojureScript 技术栈，不适合直接复用
- 非结构化数据为主，不适合结构化专利数据管理
- 表格能力弱

### 2.2 SilverBullet

**项目简介**
- 官网: https://silverbullet.md
- 技术栈：TypeScript + Deno
- 定位：可扩展的开源个人知识平台，Markdown原生
- Stars: ~6k

**核心价值与优点**
1. **Markdown原生**：页面就是纯Markdown文件，数据可移植
2. **强大的查询语言**：SilverBullet Query Language (SBQL)，可以查询页面、标签、任务等
3. **模板系统**：页面模板 + 动态查询模板
4. **插件系统**：通过Pluggable系统扩展功能
5. **实时预览**：所见即所得的Markdown编辑

**实现方式/架构特点**
- 服务端：Deno 运行时，文件系统存储
- 客户端：CodeMirror 6 编辑器，高度可定制
- 所有页面存在本地文件中，通过索引加速查询
- 插件通过 WebWorker 隔离运行

**可借鉴点**
1. ✅ **查询语言设计**：SBQL 的设计思路——用简单语法查询结构化数据
2. ✅ **模板系统**：专利Wiki页面的模板化生成可参考
3. ✅ **Markdown增强解析**：自定义标签、属性的解析机制
4. ✅ **Pluggable插件架构**：轻量级插件系统的设计

**局限性**
- 个人知识工具，不适合团队协作
- 表格、结构化数据能力弱
- Deno 生态相对小众

## 三、图谱可视化 / 图数据库前端方向

### 3.1 Cytoscape.js

**项目简介**
- 官网: https://js.cytoscape.org
- GitHub: https://github.com/cytoscape/cytoscape.js
- 定位：图论（网络）可视化与分析库
- Stars: ~10k

**核心价值与优点**
1. **功能最全面**：布局算法最多、交互最丰富、生态最完善
2. **高性能**：支持数千节点的流畅渲染
3. **布局算法丰富**：
   - 力导向：cose, cose-bilkent, cola
   - 层次：dagre, klay
   - 圆形、网格、随机、同心等
4. **扩展性强**：丰富的扩展插件（扩展布局、渲染、交互）
5. **API设计优秀**：函数式风格，链式调用，文档完善

**实现方式/架构特点**
```
Cytoscape.js 核心
├── 核心 (Core)
│   ├── 图数据模型 (elements)
│   ├── 事件系统
│   └── 动画系统
├── 渲染层 (Renderer)
│   ├── Canvas 渲染（默认，高性能）
│   └── SVG 渲染（可选，清晰度高）
├── 布局系统 (Layouts)
│   ├── 内置布局
│   └── 扩展布局（cose-bilkent, dagre等）
└── 扩展机制
    ├── 布局扩展
    ├── 渲染扩展
    └── 功能扩展
```

**可借鉴点**
1. ✅ **图数据模型设计**：节点/边的数据结构抽象
2. ✅ **布局算法选型**：专利引用/同族网络用什么布局最合适
3. ✅ **交互设计模式**：缩放、拖拽、框选、高亮相邻节点等
4. ✅ **性能优化策略**：大量节点时的渲染优化
5. ✅ **样式系统**：如何灵活定义节点/边的视觉样式

**局限性**
- 学习曲线稍陡，概念较多
- 3D、虚拟现实等高级效果弱
- 中文文档和社区资源相对少

### 3.2 AntV G6

**项目简介**
- 官网: https://g6.antv.vision
- GitHub: https://github.com/antvis/G6
- 定位：图可视化引擎，阿里蚂蚁金服出品
- Stars: ~12k

**核心价值与优点**
1. **中文生态好**：文档、教程、社区全部中文
2. **性能优秀**：支持大规模图数据渲染
3. **布局算法丰富**：内置常用布局，也支持自定义
4. **交互丰富**：内置多种交互模式（drag-canvas, zoom-canvas, drag-node等）
5. **与React/Vue集成好**：有官方封装组件

**实现方式/架构特点**
```
G6 架构
├── Graph 图对象
├── Item 元素 (Node/Edge/Combo)
├── Layout 布局
├── Behavior 交互模式
├── Plugin 插件
└── Renderer 渲染器 (Canvas/SVG/WebGL)
```

- 基于 G 绘图引擎（AntV底层渲染抽象）
- 支持 Canvas、SVG、WebGL 多种渲染模式
- 交互通过 Behavior 机制组合，可自定义

**可借鉴点**
1. ✅ **交互模式设计**：Behavior 机制——可插拔的交互组合
2. ✅ **Combo/分组功能**：专利同族分组、项目分组的可视化
3. ✅ **中文文档和示例**：学习成本低
4. ✅ **国内项目**：遇到问题更容易找到解决方案

**局限性**
- 国外社区不如Cytoscape.js活跃
- 某些高级布局算法不如Cytoscape.js丰富
- API风格偏命令式

## 四、向量数据库 / 语义搜索方向

### 4.1 Zvec

**项目简介**
- GitHub: https://github.com/alibaba/zvec
- 开发方：阿里巴巴通义实验室
- 定位：轻量级嵌入式向量数据库，"向量数据库界的SQLite"
- 协议：Apache 2.0

**核心价值与优点**
1. **嵌入式部署**：进程内运行，零配置，无需独立服务
2. **极致轻量**：pip install 即用，像普通库一样调用
3. **高性能**：基于自研Proxima引擎，毫秒级检索，支持10亿级数据
4. **功能完整**：
   - 支持多种索引类型（HNSW, IVF_FLAT, IVF_PQ, DiskANN）
   - 标量过滤 + 向量检索（混合检索）
   - MVCC 事务支持
   - LSM-Tree 存储引擎
5. **Python & Node.js 双SDK**

**实现方式/架构特点**
```
Zvec 五层架构
├── API 层：Collection / Schema / CRUD / Query
├── 计算引擎层：向量计算 + 标量过滤 + 混合检索
├── 索引层：HNSW / IVF / PQ / DiskANN 四选一
├── 存储引擎层：LSM-Tree + MVCC事务
└── 持久化层：本地文件系统
```

**Python API 示例**
```python
import zvec

# 创建集合
collection = zvec.create_collection(
    path='./patent_db',
    schema={
        'name': 'patents',
        'vectors': {
            'embedding': {'dim': 1536, 'index_type': 'hnsw'}
        },
        'scalars': {
            'patent_id': 'string',
            'category': 'string',
            'filing_date': 'int64'
        }
    }
)

# 插入向量
collection.insert({
    'embedding': [0.1, 0.2, ...],
    'patent_id': 'CN101234567A',
    'category': 'AI',
    'filing_date': 20250101
})

# 向量检索 + 标量过滤
results = collection.search(
    vector=[0.1, 0.2, ...],
    vector_field='embedding',
    filter='category == "AI"',
    topk=10
)
```

### 4.2 pgvector

- 定位：PostgreSQL 向量搜索扩展
- Stars: ~10k

**核心价值与优点**

- 新增 vector 数据类型
- 支持 IVFFlat 和 HNSW 索引

**SQL示例**
```sql
-- 创建扩展
CREATE EXTENSION vector;

-- 创建表
CREATE TABLE patents (
    id bigserial PRIMARY KEY,
    patent_id text,
    embedding vector(1536)
);

-- 相似查询
SELECT * FROM patents 
ORDER BY embedding <-> '[0.1,0.2,...]' 
LIMIT 10;
```

**可借鉴点**
1. ✅ **SQL+向量混合查询**：如何用SQL表达复杂的混合检索条件

3. ✅ **索引构建策略**：IVFFlat vs HNSW 的选择与调优

**局限性**
- 性能不如专业向量数据库（但万级数据量够用）
- 向量维度支持有限（最大2000维）
- 高级功能少（如稀疏向量、多向量字段）

## 九、高级功能拓展设计

> 本章探讨 PatWiki 在 MVP 之后可拓展的高级功能，包括字段级加密、权限体系、发布签审、协同编辑等。这些功能不是 MVP 必须，但在企业级应用场景中非常重要。

### 9.1 字段级加密方案

#### 9.1.1 为什么需要字段级加密

专利数据中存在不同敏感度的字段：
- **普通字段**：专利号、标题、公开日等（公开信息）
- **敏感字段**：发明人联系方式、内部成本、评估意见等
- **机密字段**：未公开的技术方案、商业秘密相关备注、谈判策略等

整库加密（SQLCipher）只能保护"文件被偷走"的场景，但无法解决内部人员越权访问的问题。字段级加密可以实现更细粒度的安全控制。

#### 9.1.2 加密算法选型

| 算法 | 特点 | 适用场景 | 推荐度 |
|------|------|---------|--------|
| **AES-256-GCM** | 对称加密，带认证，速度快，安全性高 | 大部分业务字段加密 | ⭐⭐⭐⭐⭐ |
| **AES-256-CBC + HMAC** | 对称加密，需额外MAC保证完整性 | 兼容老系统 | ⭐⭐⭐ |
| **RSA-2048/4096** | 非对称加密，公钥加密私钥解密 | 密钥交换、数字签名 | ⭐⭐⭐⭐ |
| **ChaCha20-Poly1305** | 流密码，适合移动/嵌入式设备 | 资源受限场景 | ⭐⭐⭐ |

**推荐方案**：AES-256-GCM 作为主力加密算法
- NIST 推荐的认证加密算法
- 同时提供保密性和完整性校验
- 加密速度快，硬件加速支持好
- Python 标准库 `cryptography` 原生支持

#### 9.1.3 密钥管理体系

```
密钥分层架构
┌─────────────────────────────────────────┐
│  用户主密钥 (User Master Key)            │
│  - 由用户密码派生 (PBKDF2/Argon2)       │
│  - 不存储，只在内存中存在               │
│  - 用于加密"密钥加密密钥"               │
└───────────────────┬─────────────────────┘
                    │ 加密
┌───────────────────▼─────────────────────┐
│  密钥加密密钥 (Key Encryption Key)       │
│  - 每个用户一份，加密后存储在数据库       │
│  - 用于加密"数据加密密钥"               │
│  - 轮换时只需重新加密DEK，无需重加密数据  │
└───────────────────┬─────────────────────┘
                    │ 加密
┌───────────────────▼─────────────────────┐
│  数据加密密钥 (Data Encryption Key)      │
│  - 每个加密字段/字段组一个DEK            │
│  - 加密后存储在数据库中                 │
│  - 用于实际加密业务数据                 │
└───────────────────┬─────────────────────┘
                    │ 加密
┌───────────────────▼─────────────────────┐
│  业务数据 (Business Data)                │
│  - 加密后的字段值存储在数据库           │
│  - 不同字段可以用不同DEK加密             │
└─────────────────────────────────────────┘
```

**密钥轮换策略**：
- **用户改密码**：重新加密 KEK，DEK 和数据都不用动
- **DEK 轮换**：生成新 DEK，用旧 DEK 解密数据再用新 DEK 加密，分批后台执行
- **KEK 轮换**：生成新 KEK，重新加密所有 DEK，数据不用动

#### 9.1.4 字段加密的实现方式

**方案一：应用层加密（推荐）**

```python
# 字段加密装饰器示例
class EncryptedField:
    """加密字段描述符，自动处理加密解密"""
    
    def __init__(self, field_name, encryption_level="normal"):
        self.field_name = field_name
        self.encryption_level = encryption_level  # normal / sensitive / secret
    
    def __get__(self, instance, owner):
        if instance is None:
            return self
        encrypted_value = instance.__dict__.get(self.field_name)
        if encrypted_value is None:
            return None
        return decrypt_field(encrypted_value, self.encryption_level)
    
    def __set__(self, instance, value):
        if value is None:
            instance.__dict__[self.field_name] = None
        else:
            instance.__dict__[self.field_name] = encrypt_field(value, self.encryption_level)
```

优点：
- 数据库不感知加密，只存密文 blob
- 密钥完全由应用控制，数据库泄露了也不怕
- 可以灵活控制哪些字段加密、用什么密钥

缺点：
- 加密字段无法直接在 SQL 中查询（WHERE / ORDER BY / LIKE 都不行）
- 需要应用层处理索引和查询

**方案二：数据库扩展加密（SQLCipher 增强）**

在 SQLCipher 整库加密的基础上，对敏感字段做第二层加密。

优点：
- 实现简单，整库加密已有基础
- 对应用透明

缺点：
- 粒度还是太粗，无法做到字段级权限控制
- 性能开销叠加

**方案三：可搜索加密（进阶）**

对于需要查询的加密字段，使用"盲索引"（Blind Index）方案：

```python
def create_blind_index(value: str, pepper: str) -> str:
    """创建可搜索的盲索引
    - 相同明文 + 相同 pepper → 相同哈希值
    - 无法从哈希反推明文
    - 可以用于精确匹配查询
    """
    h = hmac.new(pepper.encode(), value.encode(), hashlib.sha256)
    return h.hexdigest()
```

使用方式：
- 加密字段存储密文（AES-GCM）
- 同时存储该字段的盲索引哈希值
- 查询时，先对查询值计算盲索引，用盲索引做精确匹配
- 模糊查询（LIKE）仍然无法支持

#### 9.1.5 性能影响评估

| 操作 | 未加密 | 应用层加密（AES-GCM） | 影响 |
|------|--------|----------------------|------|
| 单条写入 | ~0.1ms | ~0.5ms | +400%（但绝对值仍很小） |
| 单条读取 | ~0.1ms | ~0.5ms | +400% |
| 批量写入（1000条） | ~10ms | ~80ms | +700% |
| 数据库文件体积 | 基准 | +10~30%（IV + 认证标签开销） | 可接受 |

**结论**：单条操作的加密解密开销在亚毫秒级，对用户体验几乎无感知。批量操作影响稍大，但可以通过异步处理优化。

#### 9.1.6 与现有架构的集成

- **MVP阶段**：只做整库加密（SQLCipher），保证数据文件安全
- **V2阶段**：引入字段级加密，先对最敏感的几个字段加密
- **V3阶段**：完善密钥管理、盲索引、审计日志

### 9.2 权限体系设计

#### 9.2.1 为什么单用户工具也需要权限

虽然 MVP 定位是本地单用户桌面应用，但：

#### 9.2.2 权限模型：RBAC + ABAC 混合

```
权限模型
┌─────────────────────────────────────────────────────────┐
│                    访问控制层                             │
├───────────────────────┬─────────────────────────────────┤
│    RBAC (角色权限)      │      ABAC (属性权限)            │
│  - 角色是权限的集合     │  - 基于属性的动态权限           │
│  - 静态分配，简单直观   │  - 灵活，适合复杂场景           │
│  - 适合粗粒度控制       │  - 适合细粒度控制               │
└───────────────────────┴─────────────────────────────────┘
```

#### 9.2.3 角色体系（RBAC）

**核心角色定义**：

| 角色 | 说明 | 典型用户 |
|------|------|---------|
| **超级管理员** | 拥有所有权限，可以管理用户、角色、系统配置 | 系统管理员 |
| **专利管理员** | 专利全生命周期管理、用户管理（普通用户）、配置管理 | 专利部门负责人 |
| **专利工程师** | 日常专利操作：录入、编辑、查询、分析、导入导出 | 专利工程师/代理人 |
| **发明人** | 查看自己的发明、提交提案、查看状态 | 研发人员 |
| **项目经理** | 项目维度查看、报表、风险跟踪 | 产品/项目经理 |
| **查看者** | 只读权限，不能修改任何数据 | 外部顾问、审计人员 |
| **访客/演示模式** | 只能看演示数据，不能碰真实数据 | 演示、试用 |

**角色继承关系**：
```
超级管理员
    └── 专利管理员
            └── 专利工程师
                    ├── 发明人
                    └── 项目经理
                            └── 查看者
                                    └── 访客
```

#### 9.2.4 权限维度

**三层权限体系**：

```
┌─────────────────────────────────────────────────┐
│  1. 功能权限（菜单/按钮级）                       │
│  - 能不能看到某个菜单、点击某个按钮               │
│  - 由角色直接控制                               │
├─────────────────────────────────────────────────┤
│  2. 数据权限（行级）                              │
│  - 能看到哪些数据（自己的/本部门的/全部的）        │
│  - 由角色 + 数据范围属性控制                     │
├─────────────────────────────────────────────────┤
│  3. 字段权限（列级）                              │
│  - 能看到一条数据的哪些字段                       │
│  - 敏感字段只有特定角色能看                       │
│  - 机密字段需要额外认证（二次密码/指纹）          │
└─────────────────────────────────────────────────┘
```

**功能权限示例**：

| 功能模块 | 查看者 | 发明人 | 专利工程师 | 专利管理员 | 超级管理员 |
|---------|--------|--------|-----------|-----------|-----------|
| 专利列表 | ✅ | 仅自己的 | ✅ | ✅ | ✅ |
| 新建专利 | ❌ | ❌ | ✅ | ✅ | ✅ |
| 编辑专利 | ❌ | ❌ | ✅ | ✅ | ✅ |
| 删除专利 | ❌ | ❌ | ⚠️（需审批） | ✅ | ✅ |
| 导入Excel | ❌ | ❌ | ✅ | ✅ | ✅ |
| 导出Excel | ⚠️（加水印） | ⚠️（仅自己的） | ✅ | ✅ | ✅ |
| 用户管理 | ❌ | ❌ | ❌ | ✅ | ✅ |
| 系统配置 | ❌ | ❌ | ❌ | ❌ | ✅ |

**数据权限示例（行级）**：

| 数据范围 | 说明 |
|---------|------|
| **个人数据** | 只能看自己负责的专利（assignee = 当前用户） |
| **部门数据** | 可以看本部门所有专利 |
| **项目数据** | 可以看参与项目的所有专利 |
| **全部数据** | 可以看所有专利 |

**字段权限示例（列级）**：

| 字段 | 查看者 | 发明人 | 专利工程师 | 专利管理员 |
|------|--------|--------|-----------|-----------|
| 专利号 | ✅ | ✅ | ✅ | ✅ |
| 标题 | ✅ | ✅ | ✅ | ✅ |
| 发明人 | ✅ | ✅ | ✅ | ✅ |
| 内部成本 | ❌ | ❌ | ⚠️（脱敏） | ✅ |
| 评估意见 | ❌ | ❌ | ✅ | ✅ |
| 机密备注 | ❌ | ❌ | ⚠️（需二次认证） | ✅ |

#### 9.2.5 单用户场景下的权限实现

即使只有一个用户，权限系统也有意义：

1. **演示模式**：给客户/同事演示时，切换到访客角色，只显示演示数据，不暴露真实数据
2. **防误操作**：日常使用用"专利工程师"角色，关键操作（删除、批量修改）需要切换到管理员角色并输入密码
3. **多身份切换**：用户可能同时是"专利工程师"和"项目经理"，不同视角看数据

**实现方式**：
- 本地用户表只有一个用户，但可以有多个角色
- 角色切换需要密码验证
- 操作审计日志记录当前角色

#### 9.2.6 与字段加密的联动

字段权限和字段加密是天然的组合：

- **能看的字段**：自动解密显示
- **不能看的字段**：显示 `******` 或"无权限"
- **机密字段**：需要额外认证（二次密码/指纹）后才临时解密，一段时间后自动重新加密
- **审计日志**：所有解密操作都有记录（谁、什么时候、看了哪个字段）

### 9.3 文件发布签审流程设计

#### 9.3.1 应用场景

专利工作中大量文档需要审批：
- 发明提案评审
- 交底书审核
- 申请文件定稿审核
- 审查意见答复审核
- 年费缴纳审批
- 合同/协议审核

#### 9.3.2 核心概念

```
发布签审核心模型
┌─────────────────────────────────────────────────┐
│  审批流（Workflow）                              │
│  - 定义审批流程的模板（几步、每步谁审批）         │
│  - 可配置、可复用                               │
├─────────────────────────────────────────────────┤
│  审批实例（WorkflowInstance）                    │
│  - 某个具体文档的一次审批过程                     │
│  - 关联具体的文档和发起人                        │
│  - 有状态：审批中/已通过/已拒绝/已撤回            │
├─────────────────────────────────────────────────┤
│  审批节点（WorkflowNode）                        │
│  - 审批流中的一步                               │
│  - 有类型：会签/或签/依次审批                    │
├─────────────────────────────────────────────────┤
│  审批任务（WorkflowTask）                        │
│  - 分配给具体人的待办事项                         │
│  - 有状态：待审批/已通过/已拒绝/已转交            │
├─────────────────────────────────────────────────┤
│  审批意见（WorkflowComment）                     │
│  - 审批人的意见、附件、签名                       │
└─────────────────────────────────────────────────┘
```

#### 9.3.3 审批流类型

| 审批类型 | 说明 | 示例 |
|---------|------|------|
| **依次审批** | 按顺序一个一个审，前面的通过了才到下一个 | 部门主管 → 专利总监 → 法务 |
| **会签（AND）** | 所有人都要同意，全部通过才算通过 | 所有发明人都要确认交底书内容 |
| **或签（OR）** | 只要一个人同意就通过 | 三位专利工程师任一审核即可 |
| **条件分支** | 根据条件走不同的审批路径 | 金额 > 10万 加一级审批 |

#### 9.3.4 审批流配置示例

**发明提案评审流程**：

```
发明人提交提案
     ↓
部门主管审核（确认技术方向）
     ↓
专利工程师初审（评估专利性）
     ↓
 ┌───┴───┐
 │ 条件判断 │ （技术重要度 × 商业价值）
 └───┬───┘
     ├─> 高价值 → 专利总监 + 技术总监 会签
     └─> 普通价值 → 专利工程师 直接定稿
     ↓
归档（状态变为"已立项"）
```

#### 9.3.5 版本管理与电子签名

**版本管理**：
- 每次提交审批生成一个新版本
- 审批过程中可以修改并重新提交，生成新版本
- 所有历史版本都保留，可以对比差异
- 审批通过的版本标记为"正式版本"，不可修改
- 需要修改正式版本 → 创建新版本走审批

```
版本树示例：
v1.0 (草稿)
  └─> v1.1 (提交审批)
        ├─> v1.2 (审批中修改)
        │     └─> v1.3 (审批通过 → 正式版) ✅
        └─> v1.4 (另一分支修改)
              └─> ...
```

**电子签名**：

| 签名方式 | 安全性 | 实现复杂度 | 适用场景 |
|---------|--------|-----------|---------|
| **密码确认** | 低 | 极低 | 内部一般流程 |
| **短信验证码** | 中 | 低 | 需要稍强认证的场景 |
| **数字证书（CA）** | 高 | 高 | 法律要求的正式文件 |
| **时间戳** | 中 | 中 | 证明文件存在时间 |

**推荐方案**：
- 内部审批：密码确认 + 操作日志（足够了）
- 对外正式文件：对接第三方 CA 服务（如需要）
- 核心原则：审批记录 = 操作人 + 时间 + 意见 + IP + 设备指纹

#### 9.3.6 与专利业务的结合

**审批触发点**：

| 业务动作 | 触发审批流 | 说明 |
|---------|-----------|------|
| 发明提案提交 | 提案评审流 | 评估是否申请专利 |
| 申请文件定稿 | 审核流 | 核稿、确认递交 |
| 审查意见答复 | 答复审核流 | 确认答复方案 |
| 年费缴纳 | 费用审批流 | 确认缴费金额和预算 |
| 专利放弃 | 放弃审批流 | 确认放弃理由 |
| 合同签订 | 合同审批流 | 许可/转让/质押等 |

**审批结果回写**：
- 审批通过 → 专利状态自动流转（如"提案" → "已立项"）
- 审批拒绝 → 退回修改，状态回退
- 所有审批记录挂在专利时间线上

### 9.4 协同编辑功能设计

#### 9.4.1 应用场景

专利工作中需要多人协作的场景：
- 发明人 + 专利工程师 共同撰写交底书
- 多个代理人共同处理一个复杂案件
- 审核人员在文档上批注、修改
- 团队内部对专利价值的讨论和标注

#### 9.4.2 协同编辑技术方案对比

| 方案 | 原理 | 实时性 | 冲突处理 | 实现难度 | 推荐度 |
|------|------|--------|---------|---------|--------|
| **CRDT** | 无冲突复制数据类型，自动合并 | 实时 | 自动合并，不会冲突 | 高 | ⭐⭐⭐⭐ |
| **OT** | 操作转换，Google Docs 用的方案 | 实时 | 服务端转换，需要中心节点 | 很高 | ⭐⭐⭐ |
| **锁机制** | 同一时间只能一个人编辑 | 非实时 | 简单，没有冲突 | 低 | ⭐⭐⭐⭐⭐（MVP先做这个） |
| **版本对比合并** | 各自改各自的，最后手动合并 | 异步 | 手动合并 | 中 | ⭐⭐⭐ |

#### 9.4.3 分阶段实施策略

**阶段一：悲观锁 + 版本管理（MVP 后期）**

最简单的协作方式，90% 的场景够用：

```
用户A打开编辑 → 文档被锁定（加锁人、加锁时间）
    ↓
用户B打开 → 提示"正在被A编辑中，你可以查看只读版本"
    ↓
用户A保存并关闭 → 锁释放
    ↓
用户B可以编辑了
```

补充机制：
- **锁超时**：加锁后 30 分钟无操作自动释放
- **强制解锁**：管理员可以强制释放别人的锁
- **只读模式**：别人编辑时可以看只读版本，不能改
- **版本对比**：编辑前后版本可以 diff 对比

优点：
- 实现极其简单
- 没有冲突问题
- 用户理解成本低

缺点：
- 不是实时协作
- 可能出现"占着茅坑不拉屎"的情况（有超时机制缓解）

**阶段二：评论 + 批注模式（V2）**

不需要同时改同一段内容，但可以在文档上讨论：

- 选中文本 → 添加批注
- 批注有：@提及、回复、解决、重新打开
- 批注记录：作者、时间、状态
- 文档右侧批注栏

**阶段三：实时协同编辑（V3，需要服务端）**

引入 CRDT 实现真正的实时协作：

```
技术选型建议：
- 编辑器：TipTap (基于 ProseMirror)
- CRDT 库：Yjs (成熟、社区活跃、TipTap 官方支持)
- 同步层：WebSocket
- 服务端：Node.js / Elixir / Supabase Realtime
```

```
Yjs + TipTap 架构
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  用户A编辑器 │     │  用户B编辑器 │     │  用户C编辑器 │
│  (TipTap)  │     │  (TipTap)  │     │  (TipTap)  │
└──────┬──────┘     └──────┬──────┘     └──────┬──────┘
       │  Yjs Doc          │  Yjs Doc          │  Yjs Doc
       │  共享文档状态      │  共享文档状态      │  共享文档状态
└──────┴───────────────────┴───────────────────┴──────┘
                    │       WebSocket
                    ▼
           ┌─────────────────┐
           │  同步服务器      │
           │  (Yjs WebSocket │
           │   Provider)    │
           └────────┬────────┘
                    │ 持久化
                    ▼
           ┌─────────────────┐
           │   数据库        │
           │  (Yjs update 存 │
           │   储 + 快照)    │
           └─────────────────┘
```

#### 9.4.4 操作历史与版本回溯

不管哪个阶段，操作历史都是必须的：

```
操作历史记录
┌───────────────────────────────────────────────┐
│  每条记录包含：                                 │
│  - 操作人                                      │
│  - 操作时间                                    │
│  - 操作类型（新增/修改/删除/移动）              │
│  - 修改前内容（可配置是否存完整内容）           │
│  - 修改后内容                                  │
│  - IP / 设备信息                               │
└───────────────────────────────────────────────┘
```

**版本回溯能力**：
- 可以查看任意历史版本
- 可以恢复到任意历史版本
- 可以对比任意两个版本的差异
- 重要版本可以打标签（如"v1.0 正式版"）

#### 9.4.5 本地桌面应用的协作方式

PatWiki 是本地桌面应用，协作需要特殊考虑：

**方案一：共享文件模式（最简单）**
- 数据库文件放在共享文件夹（NAS / 网盘）
- 用文件锁机制实现悲观锁
- 优点：零配置，和现有架构兼容
- 缺点：不支持实时协作，并发写入有风险

**方案二：P2P 模式（进阶）**
- 基于 CRDT 的 P2P 同步
- 每个用户本地都有完整数据
- 通过 WebRTC / 局域网发现同步
- 优点：不需要服务器，数据仍在本地
- 缺点：实现复杂，网络穿透问题

**方案三：中心服务器模式（企业级）**
- 部署一个中心服务器（可以是 Supabase）
- 所有客户端连服务器同步
- 优点：功能最完整，支持实时协作
- 缺点：需要服务器，数据不再纯本地

**推荐路线**：方案一（MVP）→ 方案三（企业版），P2P 作为可选补充

## 十、未来架构演进 — 对接 Supabase + Agent 工作模式

> 本章探讨：如果公司层面未来采用了 Supabase + Agent 的工作方式，PatWiki 如何面向未来对接，做到既不浪费当前投入，又能平滑升级。

### 10.1 背景与假设

**未来公司层面的技术栈假设**：
- 数据层：Supabase（PostgreSQL + Auth + Storage + Realtime）
- AI 层：Agent 平台（可能是公司自建或采购的 AI 代理平台）
- 应用层：各个业务系统通过 API 对接

**PatWiki 的定位变化**：
- 现在：独立的桌面工具，数据全在本地
- 未来：可以选择"本地模式"或"云同步模式"，云模式下与公司基础设施打通

### 10.2 设计原则

1. **本地优先，云端可选**：永远保留纯本地模式，云端同步是可选项，不是必须
2. **渐进式对接**：不是非黑即白，可以部分功能对接云端，部分仍在本地
3. **数据主权在用户**：用户决定哪些数据上传、哪些留在本地
4. **抽象层隔离**：通过适配层隔离具体实现，换后端不影响业务逻辑
5. **离线可用**：即使云端挂了，本地仍能正常工作，联网后再同步

### 10.3 架构分层设计

#### 10.3.1 当前架构（MVP 阶段）

```
PatWiki 桌面应用
┌───────────────────────────────────────────┐
│  React 前端                                │
├───────────────────────────────────────────┤
│  FastAPI 后端 (Python)                     │
│  - 业务逻辑                                │
│  - API 接口                                │
├───────────────────────────────────────────┤
│  本地存储层                                │
│  ├─ SQLite (主数据)                        │
│  ├─ Zvec (向量检索)                        │
│  └─ 本地文件系统 (附件)                    │
└───────────────────────────────────────────┘
         │
    本地进程内调用
```

#### 10.3.2 抽象层引入（V2 阶段）

在业务逻辑和存储层之间引入**存储适配层**，屏蔽具体存储实现：

```
PatWiki 桌面应用
┌───────────────────────────────────────────┐
│  React 前端                                │
├───────────────────────────────────────────┤
│  FastAPI 后端 (Python)                     │
│  - 业务逻辑层 (Service)                    │
│  - API 接口层                              │
├───────────────────────────────────────────┤
│  ◆ 存储适配层 ◆ (Storage Adapter)          │
│     统一接口，多种实现                      │
│  ┌─────────────────────────────────────┐  │
│  │  Interface: PatentRepository        │  │
│  │  - get_by_id()                      │  │
│  │  - list()                           │  │
│  │  - create()                         │  │
│  │  - update()                         │  │
│  │  - delete()                         │  │
│  │  - search()                         │  │
│  └─────────────────────────────────────┘  │
│           │            │                  │
│  ┌────────▼───┐  ┌─────▼──────────┐      │
│  │ SQLite实现 │  │  Supabase实现  │      │
│  │ (本地模式) │  │   (云模式)     │      │
│  └────────────┘  └────────────────┘      │
└───────────────────────────────────────────┘
```

**关键设计点**：
- 业务逻辑层只依赖接口，不依赖具体实现
- 切换存储后端只需要改配置，不用改业务代码

### 10.4 与 Supabase 的对接点

#### 10.4.1 数据层对接

| 功能 | 本地实现 | Supabase 实现 | 对接方式 |
|------|---------|--------------|---------|
| **结构化数据** | SQLite | PostgreSQL | 通过 Repository 适配层，表结构尽量保持一致 |
| **向量检索** | Zvec | pgvector | 抽象 VectorStore 接口，两种实现 |
| **文件存储** | 本地文件系统 | Supabase Storage | 抽象 Storage 接口 |
| **用户认证** | 本地密码 | Supabase Auth | 本地模式用本地账号，云模式用 Supabase Auth |
| **实时同步** | 不需要 | Supabase Realtime | 云模式下开启实时订阅 |

**数据模型迁移策略**：
- 数据库 schema 设计时就考虑 PostgreSQL 兼容性
- 尽量使用标准 SQL，少用 SQLite 特有语法
- 字段类型选择 PG 和 SQLite 都支持的类型
- Alembic 迁移脚本同时支持两种数据库

#### 10.4.2 认证与权限对接

```
认证模式切换
┌─────────────────────────────────────────────────┐

│  - 本地用户表                                    │
│  - 密码 / Windows 凭据 / 指纹                    │
│  - 本地角色权限体系                              │
└─────────────────────────────────────────────────┘
                         │
                   切换模式（用户选择）
                         ↓
┌─────────────────────────────────────────────────┐
│  云同步模式                                      │
│  - Supabase Auth (SSO 单点登录)                  │
│  - 公司统一账号体系                               │

└─────────────────────────────────────────────────┘
```

**企业级 SSO 对接**：
- Supabase Auth 支持 SAML/OIDC
- 可以对接公司的 AD/LDAP
- 用户不用单独注册，用公司账号登录

### 10.5 数据同步策略

#### 10.5.1 同步模型

```
数据分类与同步策略
┌─────────────────────────────────────────────────┐
│  类别一：纯本地数据（不上云）                     │
│  - 用户配置、偏好设置                            │
│  - 未公开的机密专利数据                          │
│  - 草稿、未完成的工作                            │
│  🔒 永远只在本地，不同步到云端                    │
├─────────────────────────────────────────────────┤
│  类别二：云同步数据（双向同步）                   │
│  - 已公开的专利基础信息                          │
│  - 团队协作的文档                                │
│  - 评论、批注、讨论                              │
│  🔄 双向实时同步，本地和云端都有                  │
├─────────────────────────────────────────────────┤
│  类别三：云端只读数据（只拉不取）                 │
│  - 公司级的基准数据（如费用标准、模板）           │
│  - 其他系统同步过来的数据                        │
│  📥 从云端拉取到本地缓存，本地只读                │
└─────────────────────────────────────────────────┘
```

#### 10.5.2 冲突解决

双向同步必然有冲突，需要明确的冲突策略：

| 冲突类型 | 解决策略 | 说明 |
|---------|---------|------|
| **同时修改同一字段** | 最后写入胜出 (LWW) + 人工确认 | 简单高效，关键数据可提示用户确认 |
| **同时修改不同字段** | 自动合并 | 字段级合并，互不影响 |
| **一端删除一端修改** | 删改为先 + 回收站 | 删除操作优先，但可以从回收站恢复 |
| **版本分叉** | 自动创建分支 + 人工合并 | 类似 Git 的分支思路，高级功能 |

**实现参考**：可以借鉴 Supabase Realtime + CRDT 的思路，或者用简单的版本号 + 时间戳机制。

#### 10.5.3 离线队列

网络断开时，本地操作暂存到队列，联网后自动同步：

```
离线队列机制

│        冲突检测与合并                   │
└─────────────────────────────────────────┘
```

### 10.6 升级路径规划

#### 阶段一：打好基础（MVP 阶段）

现在就做的准备工作：
1. ✅ **抽象存储层接口**：Repository 模式，不要在业务逻辑里直接写 SQL
2. ✅ **数据库 schema 兼容 PG**：尽量用标准 SQL，方便未来迁移
3. ✅ **RESTful API 设计**：后端 API 设计成标准 REST，未来可以直接暴露给 Agent
4. ✅ **操作日志与版本历史**：所有数据变更都有历史，为同步做准备
5. ✅ **配置化开关**：功能特性通过配置开关控制，方便灰度

#### 阶段二：云端同步试点（V2 阶段）

当有团队协作需求时：
1. 引入 Supabase 作为云端存储
2. 实现基础的双向同步（专利基础数据）
3. 实现 Supabase Auth 登录
4. 简单的团队协作功能（评论、共享）

#### 阶段三：深度集成（V3 阶段）

当公司级 Agent 平台上线时：
1. PatWiki 作为 Agent 的知识源工具接入
2. 实时协同编辑（基于 Supabase Realtime + Yjs）
3. 完整的审批流、工作流对接
4. 与 OA/ERP 等其他系统打通

#### 阶段四：混合智能体（远期）

终极形态：
1. 本地 AI + 云端 Agent 混合调度
2. 敏感数据本地处理，非敏感数据云端处理
3. Agent 可以调用 PatWiki 的能力，PatWiki 也可以调用 Agent 的能力
4. 形成"专利知识智能体"

### 10.7 风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| **数据安全** | 云端数据泄露 | 端到端加密，敏感数据不上云，用户可控 |
| **网络依赖** | 没网就用不了 | 本地优先，离线可用，联网后同步 |
| **迁移成本** | 从本地迁云端太麻烦 | 早做抽象层，schema 设计时就考虑兼容 |
| **厂商锁定** | 绑定 Supabase 了怎么办 | 抽象层隔离，理论上可以换成其他 BaaS 平台 |
| **数据一致性** | 同步冲突导致数据错乱 | 完善的冲突检测、版本历史、回滚机制 |
| **性能下降** | 云同步变慢了 | 本地缓存，读写本地，后台异步同步 |

## 十一、总结与建议

### 11.1 对当前项目的建议

1. **MVP 阶段不碰 Supabase**：聚焦本地桌面体验，SQLite + Zvec + FastAPI 完全够用
2. **提前做好抽象**：存储层、认证层、向量检索都用接口隔离，为未来留好口子
3. **字段加密先做简单版**：SQLCipher 整库加密先做着，字段级加密 V2 再考虑
4. **权限系统 MVP 只做角色骨架**：不用太复杂，但数据模型要设计好，方便后期扩展
5. **协同编辑从悲观锁开始**：简单实用，90% 场景够用，后期再升级

### 11.2 对未来的展望

PatWiki 的演进路线应该是：

```
本地单用户工具
     ↓
本地多角色 + 简单协作
     ↓
可选云同步 + 团队协作
     ↓
深度集成公司基础设施 (Supabase + Agent)
     ↓
专利知识智能体
```

每一步都是渐进的，每一步都是可独立使用的。永远保留纯本地模式，让用户选择。
